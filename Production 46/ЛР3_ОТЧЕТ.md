## 1. Цель работы

## 2. Анализ и выделение микросервисов

### 2.1. Текущая монолитная архитектура

```
IT School API (Монолит)
├── Users & Auth
├── Courses & Lessons
├── Students & Progress
├── Groups & Schedule
├── Events
├── News
├── Products
├── Notifications
└── Points History
```

### 2.2. Выделенные микросервисы

На основе анализа предметной области выделены следующие микросервисы:

**1. Main Backend Service** (Основной сервис)
- Users (пользователи)
- Students (студенты)
- Teachers (преподаватели)
- Courses (курсы и уроки)
- Groups (группы)
- Schedule (расписание)
- Events (события)
- News (новости)
- Products (товары)
- Points History (история баллов)

**Порт:** 8080  
**База данных:** school_site_db (PostgreSQL на порту 5435)

**2. Notification & Email Service** (Сервис уведомлений)
- Email отправка
- Email шаблоны
- Очередь email сообщений
- История отправленных уведомлений

**Порт:** 9090  
**База данных:** email_db (PostgreSQL на порту 5436)

**3. MinIO Service** (Файловое хранилище)
- Хранение файлов
- Фотографии пользователей
- Фотографии курсов и новостей
- HTML файлы уроков
- Файлы домашних заданий

**Порты:** 9000 (API), 9001 (Console)

### 2.3. Принципы разделения

1. **Bounded Context** - каждый сервис имеет свою область ответственности
2. **Database per Service** - отдельная БД для каждого сервиса
3. **Single Responsibility** - один сервис = одна бизнес-задача
4. **Loose Coupling** - минимальные зависимости между сервисами

## 4. Межсервисное взаимодействие

### 4.1. HTTP REST взаимодействие

**Main Backend → Notification Service**

Когда требуется отправить email:

```python
# Main Backend: apps/notification/services/email_service.py
import httpx

class EmailNotificationService:
    def __init__(self, email_service_url: str):
        self.base_url = email_service_url  # http://email-app:9090
    
    async def send_welcome_email(self, user_email: str, username: str):
        """Отправка приветственного письма через Email Service"""
        async with httpx.AsyncClient() as client:
            response = await client.post(
                f"{self.base_url}/api/emails/send",
                json={
                    "to": user_email,
                    "subject": "Добро пожаловать в IT School!",
                    "template": "welcome",
                    "data": {
                        "username": username
                    }
                }
            )
            return response.json()
    
    async def send_lesson_reminder(self, student_email: str, lesson_title: str, scheduled_time: str):
        """Напоминание о предстоящем уроке"""
        async with httpx.AsyncClient() as client:
            response = await client.post(
                f"{self.base_url}/api/emails/send",
                json={
                    "to": student_email,
                    "subject": f"Напоминание: {lesson_title}",
                    "template": "lesson_reminder",
                    "data": {
                        "lesson_title": lesson_title,
                        "scheduled_time": scheduled_time
                    }
                }
            )
            return response.json()
```

**Использование в бизнес-логике:**

```python
# Main Backend: apps/users/use_cases/create_user.py
class CreateUserUseCase:
    def __init__(
        self,
        user_service: UserService,
        email_service: EmailNotificationService
    ):
        self.user_service = user_service
        self.email_service = email_service
    
    async def execute(self, data: UserCreate):
        # Создание пользователя
        user = await self.user_service.create_user(data)
        
        # Асинхронная отправка приветственного email
        await self.email_service.send_welcome_email(
            user.email,
            user.username
        )
        
        return user
```

### 4.2. MinIO интеграция

**Загрузка файлов:**

```python
# Main Backend: apps/users/services/photo_service.py
from minio import Minio
from uuid import uuid4

class PhotoService:
    def __init__(self):
        self.minio_client = Minio(
            "minio:9000",
            access_key="minioadmin",
            secret_key="minioadmin",
            secure=False
        )
        self.bucket = "user-photos"
    
    async def upload_user_photo(self, user_id: str, file: UploadFile):
        """Загрузка фотографии пользователя в MinIO"""
        file_name = f"{user_id}/{uuid4()}.{file.filename.split('.')[-1]}"
        
        # Загрузка в MinIO
        self.minio_client.put_object(
            self.bucket,
            file_name,
            file.file,
            length=-1,
            part_size=10*1024*1024,
            content_type=file.content_type
        )
        
        file_path = f"http://minio:9000/{self.bucket}/{file_name}"
        
        return {
            "file_name": file_name,
            "file_path": file_path
        }
    
    async def get_user_photo_url(self, file_path: str):
        """Получение presigned URL для фото"""
        return self.minio_client.presigned_get_object(
            self.bucket,
            file_path,
            expires=timedelta(hours=1)
        )
```

## 5. Docker Compose конфигурация

### 5.1. docker-compose.yml

**Файл:** [docker-compose.yml](Production 46/docker-compose.yml)

```yaml
services:
  # База данных для Main Backend
  db:
    image: postgres:17-bookworm
    container_name: school_site_db
    environment:
      POSTGRES_DB: '${DB_DATABASE}'
      POSTGRES_USER: '${DB_USER}'
      POSTGRES_PASSWORD: '${DB_PASSWORD}'
    restart: unless-stopped
    ports:
      - "5435:5432"
    networks:
      - school_site-network
    volumes:
      - school_site_data:/var/lib/postgresql/data

  # База данных для Email Service
  email_db:
    image: postgres:17-bookworm
    container_name: email_db
    environment:
      POSTGRES_DB: '${DB_DATABASE}'
      POSTGRES_USER: '${DB_USER}'
      POSTGRES_PASSWORD: '${DB_PASSWORD}'
    restart: unless-stopped
    ports:
      - "5436:5432"
    networks:
      - school_site-network
    volumes:
      - email_data:/var/lib/postgresql/data

  # Основной бэкенд сервис
  main-app:
    build:
      context: .
      dockerfile: ./docker/backend/Dockerfile
    container_name: school_site_backend
    restart: unless-stopped
    expose:
      - 8080
    ports:
      - "8080:8080"
    entrypoint: /docker/entrypoint.sh
    depends_on:
      - db
      - minio
    environment:
      EMAIL_SERVICE_URL: "http://email-app:9090"
      MINIO_URL: "minio:9000"
    volumes: 
      - ./backendITSchool/.:/app/
    networks:
      - school_site-network
  
  # Email сервис
  email-app:
    build:
      context: .
      dockerfile: ./docker/email/Dockerfile
    container_name: school_site_email_notifier
    restart: unless-stopped
    expose:
      - 9090
    ports:
      - "9090:9090"
    entrypoint: /docker/entrypoint.sh
    depends_on:
      - email_db
    volumes: 
      - ./backendNotificationEmailService/.:/app/
    networks:
      - school_site-network

  # MinIO файловое хранилище
  minio:
    image: minio/minio:latest
    container_name: minio_storage
    ports:
      - "9000:9000"    # API
      - "9001:9001"    # Console
    volumes:
      - minio_data:/data
    environment:
      MINIO_ROOT_USER: minioadmin
      MINIO_ROOT_PASSWORD: minioadmin
    command: server --console-address ":9001" /data
    networks:             
      - school_site-network

networks:
  school_site-network:
    driver: bridge

volumes:
  school_site_data:
  email_data:
  minio_data:
```

### 6.1. Docker Network

```bash
# Проверка сети
```

Внутри Docker Compose сервисы находят друг друга через DNS:

## 7. Преимущества микросервисной архитектуры

### 7.1. Независимое развертывание
✅ Каждый сервис можно обновлять отдельно  
✅ Откат изменений без влияния на другие сервисы  
✅ Независимые CI/CD пайплайны

### 7.2. Масштабирование
✅ Можно масштабировать отдельные сервисы  
✅ Email service может иметь больше инстансов при высокой нагрузке  
✅ Независимое управление ресурсами

### 7.3. Технологическое разнообразие
✅ Разные технологии для разных задач  
✅ Специализированные базы данных  
✅ Оптимальный выбор инструментов

### 7.4. Отказоустойчивость
✅ Падение одного сервиса не роняет всю систему  
✅ Graceful degradation  
✅ Circuit breaker паттерны

### 7.5. Организация команд
✅ Разные команды работают над разными сервисами  
✅ Четкие границы ответственности  
✅ Упрощение onboarding
